Security Information and Event Management
# Log Sources and Analysis Challenges

Devices in a network continuously generate logs, acting as **log sources**. These logs provide a critical trail for identifying malicious activity and general troubleshooting. 

## 1. Categories of Log Sources

### [[Host-Centric Logs]]
Capture events that occur *within* or are directly related to a specific host device (e.g., Windows PCs, Linux endpoints, servers).
* **Examples:** * User accessing a file
  * User authentication attempts
  * Process executions
  * Adding/editing/deleting a registry key
  * PowerShell execution

### [[Network-Centric Logs]]
Capture events generated when hosts communicate with each other or the internet. These are generated by infrastructure devices (e.g., Firewalls, IDS/IPS, routers).
* **Examples:** * SSH connections
  * File access via FTP
  * General web traffic
  * VPN access to company resources
  * Network file sharing activity

---

## 2. Challenges of Raw Log Analysis

Relying on raw logs scattered across a network presents significant investigative hurdles:

* **Numerous Log Sources:** Hundreds of events are generated per second across multiple devices, making individual examination extremely tedious.
* **No Centralisation:** Analysts must manually connect to each machine (via SSH, RDP, etc.) to view its logs, heavily wasting investigation time.
* **Limited Context:** Individual logs rarely tell the whole story. Without correlation, a normal-looking event (like accessing a file) might hide part of a larger attack (like lateral movement after a compromise).
* **Limited Analysis:** The sheer volume of logs makes manual analysis nearly impossible, guaranteeing that human analysts will miss important events.
* **Format Issues:** Different log sources generate logs in various formats, forcing analysts to learn and parse multiple distinct structures.

# [[SIEM]] (Security Information and Event Management)

A SIEM is a security solution that resolves log analysis challenges by collecting logs from various sources, standardising their format, correlating them, and detecting malicious activities using defined rules.

## Core Features of a SIEM

### 1. Centralised Log Collection
* **How it works:** Pulls logs from all network sources (endpoints, servers, firewalls) using lightweight agents or APIs.
* **Benefit:** Centralises data in one platform, eliminating the need to connect to each machine individually.

### 2. [[Normalisation]] and [[Parsing]]
Raw logs come in many different formats and sizes. SIEM solves this by:
* **Parsing:** Breaking down a raw log into several distinct, understandable fields.
* **Normalisation:** Converting all parsed logs from various sources into one consistent, unified format.

### 3. [[Correlation]] of Logs
Individual logs often look harmless on their own. 
* **How it works:** SIEM connects and finds relationships between logs from different sources to identify malicious patterns.
* **Example:** 1. User logs in via VPN from an unusual IP.
    2. User accesses documents on a shared drive.
    3. User executes a PowerShell script.
    4. System makes an outbound connection.
    * *Correlation Result:* Flags a potential data exfiltration event resulting from compromised VPN credentials.

### 4. Real-time Alerting
* **How it works:** Uses built-in default rules and custom-created detection rules to monitor for threats.
* **Action:** When log data satisfies the rule conditions, an alert is triggered, notifying analysts to investigate within the SIEM.

### 5. Dashboards and Reporting
Dashboards present the normalised, ingested data as actionable insights. 
* **Capabilities:** Both default and custom dashboard creation.
* **Common Data Displayed:** Alert highlights, system notifications, health alerts, failed login attempts, event ingest counts, rules triggered, and top domains visited.


> *Note: Other features of a SIEM include threat intelligence feed integration, extensive data retention, and powerful searching capabilities.*

# Log Sources and Ingestion

Every device in a network generates logs when activities occur. These **Log Sources** must feed their data into a [[SIEM]] for monitoring, which is achieved through various **Log Ingestion** methods.

## 1. Common Log Sources

### Windows Machines
* Records every event, viewable through the built-in **[[Event Viewer]]**.
* Assigns a unique Event ID to each log activity for easy tracking.
* Endpoint logs are forwarded to the SIEM for centralized monitoring.

### Linux Machines
Stores events, errors, and warnings in specific file directory locations:
* `/var/log/httpd` or `/var/log/apache`: Web server HTTP requests, responses, and errors.
* `/var/log/cron`: Events related to automated [[Cron]] jobs.
* `/var/log/auth.log` and `/var/log/secure`: Authentication-related events.
* `/var/log/kern`: Kernel-related events.

### Web Servers
* Generates logs for all incoming and outgoing requests.
* Essential for identifying potential web application attacks.

## 2. Log Ingestion Methods

SIEM solutions use different mechanisms to collect (ingest) data from the network's log sources:

* **Agent / Forwarder:** A lightweight tool installed directly on the endpoint (e.g., Splunk Forwarder) that captures and sends important logs to the SIEM server.
* **[[Syslog]]:** A widely used standard protocol used to collect and forward real-time data from systems (like web servers or databases) to a centralized destination.
* **Manual Upload:** Allowing users to upload offline log data directly into the SIEM (like Splunk or ELK) for quick parsing, normalization, and analysis.
* **Port-Forwarding:** The SIEM is configured to listen on a specific port, and endpoints are configured to forward their log data directly to that listening port.
